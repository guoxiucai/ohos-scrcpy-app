# 一、 项目规划
设计一个针对OpenHarmony设备远程投屏和HDC调试工具，能够将OpenHarmony设备的屏幕实时镜像到计算机，并提供设备控制功能。
分为客户端和服务端，客户端运行在个人开发电脑上（windows/mac/linux），服务端运行在OpenHarmony设备上，客户端和服务端通过USB或网络 HDC连接。

1. 客户端能力设想
 - P0：UI基于Flutter实现（支持跨平台），单窗口应用
 - P0：HDC设备发现，支持多个设备列表切换
 - P0：接收服务端的H264视频流并实时渲染显示OpenHarmony设备屏幕，自动调整显示尺寸，保持原始比例
 - P0：支持可视化的hap应用安装
 - P0：支持在屏幕渲染区域发送点击，长按、鼠标滑动、右键、鼠标滚轮控制命令
 - P1：支持文本输入（实时键盘输入优先，兜底在客户端预留一个输入框）--todo
 - P1：支持基本的音量设备调节按钮，设备亮度调节按钮
 - P2: 虚拟一个终端界面，用户可以在此模拟发送hdc命令和查看响应（或者嵌入一个真实的终端）
 - 客户端工程代码放在 scrcpy_client_flutter目录，本机 Flutter版本是3.22.1
 - 最终编译产物，mac是dmg文件，windows是exe文件，linux是deb文件，优先支持mac和windows，linux可以稍后。


2. 服务端能力设想
 - P0: 屏幕采集和H264编码
 - P0: 视频流发送
 - P0: 随设备开机自启动（keepAlive）
 - 服务端是OpenHarmony系统服务应用，最低支持的API版本是15，支持ServiceExtensionAbility和keepAlive
 - 服务端代码放在 scrcpy_server目录，已经创建好系统应用模板工程

3. 客户端和服务端通过自定义TCP协议进行通信，协议参考：
 - 数据包格式：4字节包类型 + 4字节数据长度 + 数据内容
 - 包类型：心跳、视频帧、配置信息


# 任务2 如何使用ServiceExtension自启动
 module.json5中mainElement需要调整为与ServiceExtensionAbility名字保持一致，这样当服务自动时就是执行的ServiceExtension逻辑。 MainAbility中不需要有什么业务逻辑，当ServiceExtension运行时，自动完成服务端的所有能力。

# 优化
 - 连接断开后（如客户端主动断开或异常断开），服务端立即停止截屏/编码/传输，等待下次连接（现在有个问题，当客户端断开后，再次连接，不会渲染视频了）
 - 客户端增加30s心跳超时断开逻辑，服务端也增加30s心跳超时逻辑，断开连接
 - 客户端设备列表增加上设备的name，格式：[name]+id，其中name通过hdc shell param get const.product.name获取
 - 设备连接成后，列表右侧的指示文案为"已连接，等待视频流...", 当第一帧视频渲染后，指示文案为："已连接，镜像中"

# 安装/卸载应用
1. 已有“安装应用”区域：选择hap，安装成功直接弹窗告知结果成功（不需要现在的提示原始信息），安装失败弹窗错误信息，HDC安装时使用强制模式
2. 新增“卸载应用”区域（位于安装应用下面）： 支持用户通过下拉列表选择应用包名，也支持输入关键字自动匹配列表中的项。服务端已经增加获取安装应用列表系统权限，下拉列表的所有应用需要通过服务端获取（通过hdc命令，或者自定义协议让服务端获取， 这里需要调研下那个方案更合适）；应用列表需要过滤掉具有不可移除属性的应用包名

# 卸载应用优化
1. 需要两个区域，
 - 输入区：输入包名/选择应用，根据用户输入的关键字，自动匹配下拉列表区的应用包名。这个区域不需要依赖连接状态，不连接时用户也可以手动输入包名执行卸载。
 - 设备应用下拉列表区：连接后根据协议获取服务端设备应用列表（仅显示包名即可），这部分与上面的输入区联动，eg：用户输入com, 则下来列表仅过滤包含关键词com的。此部分依赖连接状态，让未连接时，此部分为空，但不影响用户自己的输入区手动输入。
 - 卸载按钮，不依赖连接状态。
2. 用户选择某个应用包名（或手动输入），卸载成功后，应用下拉列表移除该item, 同事清空输入区，然后弹窗提示卸载成功；若失败弹窗提示失败并带上错误信息。

# windows花屏问题继续
回过头继续看花屏问题，此前你尝试给过这个方案，我拒绝了。
1. 丢帧策略：队列满时不再逐个丢弃 P 帧（会断裂参考帧链），而是清空整个队列并设置 wait_keyframe_ 标志，丢弃所有后续 P 帧直到下一个关键帧到来——保证解码器永远从完整关键帧开始。
2. MFT 状态重置：关键帧到来时先 flush MFT，清除内部残留的旧参考帧状态，确保关键帧解码不受之前丢帧的影响。              
这不是好的解决措施， 现在画面变化后会直接卡住，一直等好久画面才会恢复新的，因为服务端的关键帧间隔很大的。现在队列是多大？其实可以适当增大队列（适当延时大一点影响不是很大），如果真的是增加队列后队列也满了，应该同控制消息让服务的不再继续编新的图像了（丢帧需要在编码之前丢，编出来之后的都不能再丢了）
